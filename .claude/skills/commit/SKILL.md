# commit

コミットメッセージの「なぜ」は**発明せず転記する**。人間の承認はコミット対象の diff に
対して得る(役割分担プロトコル通り)が、メッセージの質は以下のガードレールで担保する。

## 1. 前提確認

- コミット対象の diff についてユーザーの承認を得ているか(計画駆動の作業では必須)
- 1コミット = 1論理単位になっているか

## 2. ステージング(厳密に)

- **対象ファイルをパス指定で `git add`** する。`git add -A` や `git add .` は使わない
- `git status --short` で、ステージが承認範囲と一致し、無関係な変更(作業ツリーに
  同居する未承認の変更・生成物)が混入していないことを確認する

## 3. メッセージの作成

### type と scope(閉じた語彙から選ぶだけ)

CLAUDE.md の規約に従う。判定は「外から見える挙動が増える・変わるか」の1問のみ。

### 「なぜ:」行(転記元を特定してから書く)

| 作業の種類 | 理由の転記元 |
|---|---|
| 計画駆動(run-plan-step 等) | 計画ファイルの該当ステップの目的 + 完了報告の「計画外の判断事項」 |
| スキル経由の定型作業 | スキルが収集した入力(依頼内容・出典など) |
| 単発の依頼 | ユーザーの依頼文の「〜したい」の部分 |

転記元が特定できない場合のみ自分の言葉で書き、完了報告でその旨を明示する。

### diff ブラインド自己検査(コミット前に必ず)

**本文だけを読み直して**次の2問に答える。答えられなければ書き直す:

1. 何がきっかけでこの変更をしたか分かるか
2. その内容は diff を見れば分かること(what の再述)になっていないか

## 4. 形式の自己チェック(機械的)

メッセージが次をすべて満たすことを確認してからコミットする:

- ヘッダ: `^(feat|fix|refactor|perf|test|docs|chore)(\([a-z-]+\))?!?: .+`(要約は日本語・句点なし)
- 本文に `^なぜ: .+` の行がある
- 末尾に `Co-Authored-By:` トレーラーがある

## 5. コミットと事後確認

- `git commit` 実行後、`git log -1 --stat` で対象ファイルとメッセージを確認する
- **push・PR 作成・マージはしない**(指示されても人間の作業と案内する)
- 監査(audit-plan-step)でメッセージの指摘を受けた場合、未 push であれば
  `git commit --amend` で修正してよい
