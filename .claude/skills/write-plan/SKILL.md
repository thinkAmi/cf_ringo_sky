# write-plan

計画ファイルは「設計セッションと実装セッションの間の契約書」。読む側は会話履歴を持たない
別モデルという前提で、**この文書だけで迷わず実行できる**ことを品質基準とする。

## 前提

設計決定が確定していること(docs/adr/ に記録済み)。未確定の論点が残っているなら、
計画を書き始めずに grill-with-docs での設計対話を先に提案する。

## 計画ファイルの構成(docs/YYYY-MM-<topic>-plan.md)

1. **ステータス行**(冒頭の blockquote): 現在地 + 「各フェーズ完了時に実装セッションが
   この行を更新する(更新もコミット承認プロトコルに従う)」という運用規約
2. **背景と方針**: なぜやるか・何を変えないか(不変条件)。ADR へのリンク
3. **進め方の原則**: PR 分割、1コミットの粒度、破壊的操作を最後に置く順序
4. **ブランチ戦略**: epic / 子ブランチ / マージ先。実装セッションがどのブランチから
   作業を始めるかを明示する
5. **役割分担**: 全作業に主語を割り当てる(下記チェックリスト参照)
6. **確定仕様の転記**: フォーマット・ルール表・実例。ADR への参照だけで済まさず計画に転記する
7. **ステップ詳細**: 各ステップに「作業項目」と「機械的な合格条件」のペア
8. **検証の全体設計**: ゲートの層(型 / lint / ビルド / テスト / 実行時 / 目視)と適用範囲の表
9. **不採用・先送り事項**: 検討して捨てた案と理由(実装者が「改善」として復活させるのを防ぐ)
10. **期待される最終状態**
11. **末尾の導線**: 「実行は run-plan-step、監査は audit-plan-step を使う」と明記

## 引き継ぎ品質チェックリスト(書き終えたら自己検査)

- [ ] **自己完結性**: 仕様・ルール・実例が計画に転記されている。「会話で合意した通り」への参照がない
- [ ] **参照の解決可能性**: 言及するファイル・型・スクリプトはすべてリポジトリ内に実在する
- [ ] **機械的な合否**: 全ステップに、実装者の自己評価に依らない合格条件(テスト・diff・スナップショット)がある
- [ ] **裁量の封じ込め**: データ変換は「スクリプトで機械的に生成、手書き転記禁止」と明記。
      実装者が判断に迷う**入力のギャップ**を探して潰したか
      (実例: 色定義が存在しない系譜品種の存在を計画が予見せず、実装者の無申告判断を招いた)
- [ ] **禁止リストの線引き**: 「コマンド名」でなく「本番に作用するか」で書く
      (実例: 「wrangler deploy 禁止」と書くと --dry-run のビルドゲートまで拒否される)
- [ ] **役割分担の網羅**: コミット承認・push・PR・デプロイ・DROP 等の破壊的操作・スキル起動と
      題材提供、のすべてに主語があるか。「AI がやってよいローカル操作」も明示したか
- [ ] **事故ポイントの強調**: 順序を誤ると壊れる箇所(デプロイ→DROP の順など)を「厳守」と明示
- [ ] **設計変更の扱い**: 「実装中に設計を変えたくなったら、勝手に変えず ADR 更新を提案」と書いたか

## 書いた後

計画は epic ブランチの初回コミットに含める(設計が破棄されたら計画ごと消える運用の場合)。
実装セッションへの指示文もセットで提案する:
「epic ブランチ <名前> 上で docs/YYYY-MM-<topic>-plan.md と docs/adr/ を読み、PR 1 から着手して」
