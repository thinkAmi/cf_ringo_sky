# 系譜データは D1 ではなく Git 管理の markdown 1ファイルで管理する

D1 の内容が失われると系譜図が復元できなくなる懸念から、系譜データの信頼できる唯一の情報源をリポジトリ内の markdown 1ファイルに移し、D1 の `apples` / `genealogies` テーブルは廃止する。markdown はビルド時に ringo-db Worker へ文字列としてバンドルされ、系譜の解決（曽祖父母世代までの祖先展開）は SQL の再帰 CTE に代えて TypeScript の木探索で行う。サービスバインディングのメソッドと返却 JSON の形は変えないため、ringo-web は無変更。

## Considered Options

- **(a) markdown を正として seed SQL を生成し、D1 は維持する** — 既存の CTE を温存できるが、markdown → SQL → D1 という同期パイプラインの管理が増える
- **(b) D1 を正として markdown をエクスポートする** — Git 管理済みの seed SQL と役割が重複し、喪失対策としての追加価値がない
- **(c) D1 から系譜を廃止し markdown をバンドルする（採用）** — データの実体が Git 内の1ファイルになり、環境構築から seed 投入が消える

## Consequences

- 系譜の更新に Worker の再デプロイが必要になる。ただし品種追加は従来から色定義（`appleColors.ts`）のコード変更+デプロイとセットで発生するため、運用は一本化される
- FK 制約による参照整合の担保が消えるため、パース時バリデーション（親品種の存在チェック等）とテストで代替する。テスト戦略: 祖先が全て異なる人工フィクスチャによる33フィールド検証（事例ベース）、移行期間中は `bun:sqlite` 上で旧 CTE と出力を突き合わせる差分テスト、循環系譜などの病的入力への頑健性とパーサのラウンドトリップ性はプロパティベーステスト（fast-check）で担保する
- D1 のリクエストごとの往復が消えるためレイテンシはわずかに改善する（決定要因ではない）
