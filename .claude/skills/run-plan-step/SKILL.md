# run-plan-step

計画 doc の1ステップ(PR)を実行し、ゲート通過を確認してから承認済みコミットにまとめる。
**ユーザーの承認前にコミットしない。**

## 1. プリフライト(編集を始める前に必ず)

1. **計画 doc・docs/adr/・CONTEXT.md を読む**。計画の「役割分担」「進め方の原則」節が
   このスキルの記述と食い違う場合は計画 doc を優先する
2. **ステータス行で現在地を確認**し、着手するステップが順序制約に合っているか確かめる
3. **ブランチを確認する**。epic ブランチ上にいるなら、
   `feature/<epic名>_<ステップ略称>` の子ブランチを**作ってから**編集を始める
   (実例: 品種マスタ移行 PR 1 では子ブランチを作らず epic 上で直接作業してしまった)
4. 今回のステップの作業項目を列挙し、**スコープ外(後続 PR の領分)に触れない**ことを宣言する

## 2. 実装

- 計画に書かれた作業のみ行う。計画外の変更が必要になったら、黙って進めず報告して承認を得る
- **計画の入力にない値を自分で決める場面**(データのギャップ埋め・命名・閾値など)が生じたら、
  その場で止まらなくてよいが、必ず記録して完了報告の「計画外の判断事項」で申告する
  (実例: 色定義が存在しない系譜品種3つに Red を付与した判断が、無申告のまま成果物に入っていた)
- 設計判断(ADR に書かれた内容)を変えたくなったら、勝手に変えず ADR の更新をユーザーに提案する

## 3. 機械ゲート

計画 doc の「検証の全体設計」に従う。標準は:
typecheck → `biome check` → ビルド(`wrangler deploy --dry-run` / ringo-web build)→
`bun test` → `verify.sh`(API スナップショット diff)。
すべて緑になるまで完了報告に進まない。lint・テストのエラーは行番号とメッセージを読んで修正する。

## 4. 完了報告(テンプレート)

承認を求める前に、次の4点を必ず含めて報告する:

1. **実施内容**: 何をどう変えたか(diff の要点)
2. **ゲート結果**: 各ゲートの実行結果
3. **計画外の判断事項**: 自分で決めた値・計画から逸れた点。**なければ「なし」と明記**
4. **人間レビュー依頼**: 機械検証できないデータ(読み・出典・色の選定など)の確認ポイント

## 5. 承認とコミット

- ユーザーの承認後にのみコミットする。論理単位で分割し、各コミットが機械ゲート通過状態であること
- コミットの手順・メッセージは **commit スキル**(CLAUDE.md のコミット規約 + 転記・自己チェック)に従う
- フェーズ完了時は計画 doc のステータス行を更新する(この更新も承認後のコミットに含める)

## 6. 実行してよい操作・してはいけない操作

線引きは「**本番に作用するか**」。コマンド名ではなく作用先で判断する。

- **AI が実行する**: `wrangler deploy --dry-run`、`wrangler d1 migrations apply <db> --local`、
  verify.sh、dev サーバー起動、`bun test` などローカル・ドライランの一切
- **実行しない(指示されても人間の作業と案内する)**: push、PR 作成・マージ、
  `wrangler deploy`(--dry-run なし)、`wrangler d1 migrations apply <db> --remote`
