# 品種マスタ移行計画(2026-07)

> ステータス: 未着手(計画確定日: 2026-07-08)
>
> 各フェーズ(PR マージ・スキル実証・リリース)の完了時に、実装セッションはこのステータス行を更新すること(例: 「PR 1 マージ済み(日付)」)。更新自体も通常のコミット承認プロトコルに従う。

設計判断の記録: [ADR-0001](adr/0001-genealogy-in-markdown.md) / [ADR-0002](adr/0002-variety-master-includes-colors.md) / [ADR-0003](adr/0003-variety-master-format.md)、用語集: リポジトリルートの `CONTEXT.md`。本計画はこれらの決定を実行に落としたもの。**実装中に設計判断を変えたくなった場合は、勝手に変えず ADR の更新をユーザーに提案すること。**

## 背景と方針

- D1 の内容が失われると系譜図が復元できなくなる懸念から、系譜データの source of truth を Git 管理の markdown 1ファイル(品種マスタ)に移す
- 品種マスタは色定義(`appleColors.ts` の129品種)も統合し、1品種 = 1行で色と系譜の不整合を構造的に不可能にする
- 系譜解決は SQL の再帰 CTE から TypeScript の木探索へ置き換える。**サービスバインディングのメソッド名・引数・返却 JSON の形は一切変えない**(ringo-web は無変更)
- D1 の `apples` / `genealogies` テーブルは最終的に廃止する。`feeds` テーブルと集計系メソッドは対象外
- 撤去(破壊的操作)は最後。導入 → スキル → 撤去の順で epic ブランチ内で完結させ、デプロイは epic → main マージ後に1回。**DROP の本番適用はデプロイとは別の、本番確認後の独立イベント**とする(ここまでは D1 の旧テーブルが凍結スナップショットとして残り、旧バージョンの再デプロイが即時ロールバック経路になる)

## 進め方の原則

- **1 PR = 1フェーズ**(PR 1: 導入本体 / PR 2: スキル再編 / PR 3: D1 撤去とドキュメント)。PR 間に順序依存があり、入れ替え不可
- **各 PR は機械ゲート通過が条件**: typecheck → biome check → build(`wrangler deploy --dry-run` / ringo-web build)→ `bun test` → verify.sh の API スナップショット diff ゼロ
- 品種マスタの初期データは**手書き転記ではなく変換スクリプトで機械的に生成する**(appleColors.ts と seed SQL が入力)。手書き転記はハルシネーションの入り口なので禁止。例外は「読み」列のみ(AI が記入し、人間が完了報告でレビュー)
- wrangler は devDependency のバージョンに固定して `bunx wrangler` で実行(CLAUDE.md の既存ルール)
- 本番へのマイグレーション適用(DROP)は PR 3 マージ後、かつ **新 Worker のデプロイ完了後**。順序を誤るとダウンタイムが出る

## ブランチ戦略

```
main ──── feature/variety_master(epic、長生き)
            ├─ 初回コミット: 設計ドキュメント(CONTEXT.md / docs/adr/ / 本計画)
            ├─ feature/variety_master_pr1_core      → PR base = epic
            ├─ feature/variety_master_pr2_skill     → PR base = epic
            └─ feature/variety_master_pr3_teardown  → PR base = epic
                        ↓
    main ←─ epic を1本の PR でマージ → デプロイ → 本番確認 → DROP 適用
```

- **設計ドキュメントは main ではなく epic の初回コミットに置く**。設計がうまくいかない場合は epic ごと破棄する(ADR が main に載る = 決定が実証済み、という意味論)
- 各 PR は epic を base にレビュー・マージし、順序制約(1→2→3)は epic 内で守る
- **実装セッションは必ず epic ブランチから作業を開始すること**(main には設計ドキュメントが存在しない)
- main に他の変更が入った場合は epic へ随時 merge して追従する
- epic → main のマージが「すべて終わった」の瞬間。デプロイと本番確認、その後の DROP 適用の手順は「確認と本番適用」節の通り

## 役割分担(実装セッション向け・厳守)

| 作業 | 担当 |
|---|---|
| コード変更・機械ゲート実行・完了報告 | AI(実装セッション) |
| ローカルコミット | **ユーザーが完了報告と diff を確認・承認した後にのみ** AI が実行(run-upgrade-step と同じ「機械ゲート → 完了報告 → ユーザー確認 → 1コミット」プロトコル) |
| push・PR 作成・PR マージ | 人間 |
| デプロイ・DROP の本番適用 | 人間 |

- AI は承認なしにコミットしない。**本番に作用する操作**(push / PR 作成・マージ / `wrangler deploy`(`--dry-run` なし) / `wrangler d1 migrations apply ringodb --remote`)は指示されても実行せず、人間の作業であることを案内する
- **ローカル・ドライランは AI の担当**: `wrangler deploy --dry-run`(ビルドゲート)、`wrangler d1 migrations apply ringodb --local`(PR 3 のローカル検証)、verify.sh、dev サーバー起動は機械ゲートの一部として AI が実行する
- main に変更が入った場合の epic への追従 merge は AI が実施してよい(コンフリクト解消を含む。コミットは通常通り承認後)
- 読み列・出典など機械検証できないデータの人間レビューは、コミット承認時の完了報告・diff 確認の場で行う
- コミットは論理単位で分割する(PR 1 のような大きいフェーズは複数の論理コミット可。各コミットが機械ゲート通過状態であること)

## 品種マスタの仕様(確定済み)

パス: `packages/ringo-db/data/varieties.md`

```markdown
| 表示名 | 読み | 色 | name | 花粉親 | 種子親 | 出典 |
|-------|------|-----|------|-------|-------|------|
| 紅玉 | こうぎょく | Maroon | kougyoku | unknown | unknown | |
| つがる | つがる | Goldenrod | tsugaru | kougyoku | golden_delicious | [青森りんご品種図鑑](https://example.com/) |
| シナノゴールド | しなのごーるど | Gold | | | | |
```

- 主キーは**表示名**(feeds と色定義が使う日本語の名前空間)
- name(snake_case)と親2列は系譜を持つ品種のみ埋める。**基本品種の親は空セルではなく `unknown` を明示**
- `unknown` 品種そのもの(表示名「不明」)の行は書かない。パーサが内部で合成する
- 行は「読み」の五十音順。読みはソートキーであって正式なふりがなではない(英字品種もひらがな転写: KORU → こる)
- 出典はその行の内容全般の根拠。自由記述(Web は markdown リンク、書籍は書名等)、空可、API 非公開。セル内の半角 `|` は `\|` にエスケープ

### lint ルール(バリデータ = ランタイムパーサの同一コード)

| # | ルール | エラーに含める情報 |
|---|--------|------------------|
| 1 | ヘッダ行が上記と完全一致、全行セル数が7 | 行番号・実セル数 |
| 2 | 表示名: 非空・全行で一意 | 重複する両方の行番号 |
| 3 | 読み: `^[ぁ-んー]+$`(ひらがな + 長音記号)・非空 | 行番号・違反文字 |
| 4 | 色: CSS named color 一覧に存在 | 行番号・未知の色名 |
| 5 | name: 空 or `^[a-z][a-z0-9_]*$`・非空同士で一意 | 行番号 |
| 6 | name あり行は親セル2つとも必須(実在 name or `unknown`)、name なし行は親セル空 | 行番号・対象セル |
| 7 | 参照整合: 親は他の行の name として実在すること | 行番号・未定義の品種名 |
| 8 | 循環系譜なし(自分が自分の祖先に現れない) | 循環パス |
| 9 | 読み順ソート | 行番号・期待位置 |

出典列に専用ルールはない(空でも任意テキストでも可。ルール1が構造だけ守る)。**エラーメッセージは AI の自己修正ループの入力になるため、行番号と原因を含む文面自体をテストで固定する。**

## ステップ詳細

### PR 1: 品種マスタ導入(本体) — 動作不変のリファクタリング

1. **baseline 取得**: 変更前(main)の状態で verify.sh を実行しスナップショットを確保(既存の運用があればそれに従う)
2. **変換スクリプト**: `appleColors.ts`(129品種の表示名・色)と `seed/apples_and_genealogies.sql`(17品種の name・表示名・親)を入力に `varieties.md` を生成。読み列は AI が記入(確信の持てない品種は完了報告で「読みは推定」と明示)。`unknown` 行は出力しない。スクリプトはコミット不要(生成物のみコミット)
3. **パーサ/バリデータ** `packages/ringo-db/src/varietyMaster.ts`: 純関数(markdown 文字列 → レコード配列 or 行番号つきエラー一覧)。セルは位置番号ではなく**ヘッダ名で引く**(将来の列追加を局所化)。出典列は検証(ルール1のみ)して読み飛ばし、ランタイムのデータ構造には載せない
4. **wrangler.toml** に `[[rules]] type = "Text" globs = ["**/*.md"]`、`*.md` 用の型宣言ファイルを追加
5. **系譜解決の TS 実装**: モジュールスコープで varieties.md をパースし Map 化(unknown 品種を内部合成)
   - `findGenealogyByName`: 曽祖父母世代(3世代)までの木探索 → 現行と同一の33フィールド横持ち JSON
   - `findGenealogies`: unknown を除外した6フィールド配列、**name の昇順**(現行の `orderBy(apples.name)` と一致させる)
   - `findColorName`: Map 参照に置き換え。未登録名の `'red'` フォールバックは維持(feeds の品種名は実行時データのため)
6. **`src/index.ts`**: 上記3メソッドの中身を差し替え。メソッドシグネチャ・返却形は不変。旧 CTE の SQL は差分テストへ移設
7. **テスト**(`bun:test`、fast-check を devDependency に追加):
   - lint ルール別: 1ルールにつき最低1つの違反フィクスチャ + 正常系。エラーメッセージの行番号・文面まで assert。バリデータのカバレッジ100%
   - 実物ゲート: リポジトリの varieties.md 自体が全ルールを通過すること(このテストがデプロイゲートを兼ねる)
   - 33フィールド検証: **祖先15個がすべて異なる名前の人工フィクスチャ**で全フィールドを一意な期待値と突き合わせる(詰め替え取り違えの検出。実データは祖先が重複するため不可)
   - 差分テスト: 生成データ or 現行 varieties.md を `bun:sqlite` の in-memory DB に INSERT して旧 CTE を実行し、TS 実装と出力一致を assert。ファイル冒頭に「旧 D1 実装(〜2026-07)をオラクルとする等価性検証。系譜解決の仕様を意図的に変更する PR でこのファイルごと削除してよい」とコメント
   - PBT(fast-check): ランダム系譜(循環・自己参照・長い鎖・ダイヤモンド型を含む)で「正常出力かバリデーションエラーかのどちらかに必ず収まり、無限ループしない」/ パーサのラウンドトリップ(レコード → markdown → parse → 一致)
8. **合格条件**: 機械ゲート一式 + **verify.sh diff ゼロ**(`/api/genealogies` `/api/genealogies/:名前` を含む4エンドポイント)。D1 テーブル・seed・README はこの PR では触らない

### PR 2: スキル再編

1. `.claude/skills/register-apple-variety/SKILL.md` を新設。手順:
   - 入力確認: 表示名(必須)・読み・色(CSS named color)・系譜を登録する場合は name/花粉親/種子親(不明なら `unknown`)・出典
   - 前提チェック: 親品種が未登録なら先にその親の登録を提案
   - 編集: varieties.md の読み順の正しい位置に行を挿入(既存品種は該当行を置換)。出典内の `|` は `\|` へ、複数出典は ` / ` 区切り(いずれも lint ではなく慣例)
   - 機械ゲート: `bun test` を実行し、エラーなら行番号とメッセージを読んで修正・再実行
   - 完了報告 → ユーザー確認 → 1コミット(run-upgrade-step と同じプロトコル)
   - 規約: 読みに確信が持てなければ Web 検索で確認するか「読みは推定」と申告 / 系譜登録時は出典を尋ね、なければ「出典なし」と明示 / **データファイルのみ編集し、パーサ・バリデータのコードには触れない**
2. `.claude/skills/add-apple-to-graph/` を削除(役割は品種マスタと新スキルに吸収)
3. 合格条件: 機械ゲート一式(スキルは markdown のみだが、リポジトリ状態の健全性確認として実行)

### スキル実証(PR 2 マージ後、PR 3 着手前・epic 上で実施)

register-apple-variety スキルで品種を1つ実登録し、機械ゲート〜コミットまで通ることを確認する。スキルは「編集 → `bun test` → コミット」で完結するローカル作業のため、本番デプロイなしで検証できる。

- **スキルの起動と題材の提供は人間が行う**(実装セッションのタスクではない)。登録するのは実在の品種で、その情報(表示名・読み・色・系譜・出典)はユーザーが提供する。テスト用の架空品種をでっち上げない
- 登録した品種はそのまま品種マスタに残す(revert しない)

### PR 3: D1 撤去とドキュメント

1. DROP マイグレーション追加: `genealogies` → `apples` の順で DROP(FK の向きのため)。`feeds` は残す
2. 削除: `db/schema/apples.ts` / `db/schema/genealogies.ts` / `seed/apples_and_genealogies.sql` / `appleColors.ts` のデータ定義(`findColorName` 相当は varietyMaster 側に移設済み)
3. README 更新: 環境構築から seed 投入手順を削除し、「品種の追加 = register-apple-variety スキル or varieties.md 編集 + `bun test` + デプロイ」を記載
4. CLAUDE.md 更新: seed に言及した落とし穴の書き換え(「feeds が空なら `/api/total` は `[]`」は存続)、品種マスタの存在の追記
5. 合格条件: 機械ゲート一式 + verify.sh diff ゼロ(ローカルは migrations 適用後、seed なしで系譜 API が動くこと)

### 確認と本番適用(epic → main マージ後・順序厳守)

1. epic を1本の PR で main にマージ
2. ringo-db Worker をデプロイ(`bunx wrangler deploy`)。この時点で本番はまだ DROP されていない(即時ロールバック可能)
3. 本番の系譜ページ(`/genealogies` と個別ページ)・集計グラフを目視確認
4. 問題なければ `bunx wrangler d1 migrations apply ringodb --remote` で DROP を適用(`packages/ringo-db` で実行)。**この順序を逆にしない**
5. 問題があれば DROP を適用せず、旧バージョンの再デプロイで復旧して原因調査に戻る

| 層 | 手段 | 適用範囲 |
|---|---|---|
| 1. 型 | `tsc --noEmit`(全パッケージ) | 全 PR |
| 2. lint | `biome check .` | 全 PR |
| 3. ビルド | `wrangler deploy --dry-run` / ringo-web build | 全 PR |
| 4. 単体 | `bun test`(バリデータ100% / 33フィールド / 差分 / PBT) | PR 1 以降常設 |
| 5. 実行時 | verify.sh(API ゴールデンスナップショット diff) | 全 PR |
| 6. 目視 | スキル実登録1件(epic 上・PR 3 前) / 本番の系譜ページ・集計グラフ(マージ・デプロイ後、DROP 前) | 確認フェーズ |

## 不採用・先送り事項

| 項目 | 判断 | 理由 |
|---|---|---|
| markdown → seed SQL 生成で D1 維持(案a) | 不採用 | 同期パイプラインが増えるだけ。ADR-0001 参照 |
| YAML / CSV / JSON / mermaid 形式 | 不採用 | ADR-0003 参照。検証の本丸(参照整合・循環)はどの形式でも自作になる |
| 基本品種の親を空セルで表現 | 不採用 | 親の消し間違いが silent に基本品種化する。ADR-0003 参照 |
| 出典の URL 書式 lint | 不採用 | 書籍出典を許容するため自由記述とした。ADR-0003 参照 |
| 複数系譜(1品種に複数の親子説) | 想定しない | 系譜図 UI の表示方法が定まらない。表示名一意ルールが構造的に排除 |
| 未知の列を許容する lint 等の投機的柔軟性 | 不採用 | strict lint を弱める。列追加は ADR-0003 の手順(意味論・lint・露出・スキル規約の4点)で都度行う |
| 差分テストの早期削除 | 不採用 | 維持コストほぼゼロ。系譜解決の仕様を意図的に変える最初の PR で削除(条件はテストファイルに明記) |
| feeds ↔ 品種マスタの整合(食べた品種の未登録検出) | 対象外 | feeds は実行時データでありビルド時に守れない。従来と同じ性質 |

## 期待される最終状態

- 系譜・色の source of truth が `packages/ringo-db/data/varieties.md` 1ファイル(Git 管理・GitHub で閲覧可能な品種一覧表)になる
- D1 は `feeds` テーブルのみ。ローカル環境構築から seed 投入が消える
- 品種の追加・更新は register-apple-variety スキル(編集 → 機械ゲート → 確認 → 1コミット)に一本化
- バリデータのカバレッジ100% + 旧実装との等価性検証 + PBT による頑健性の網が常設される
