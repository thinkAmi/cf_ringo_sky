# フィードは生のまま保存し、集計時に品種マスタの表示名で絞り込む

Bluesky の投稿から抽出したフィードは、品種マスタとの突き合わせなしに生のまま feeds テーブルへ保存されており、`foo` のようなマスタ未登録の名前もそのままグラフに表示されていた。これを防ぐため、突き合わせは**読み取り時のみ**行う: ringo-db の集計 RPC（`calculateTotalByName` / `calculateTotalByNameAndMonth`）が SQL 実行後の TS 層で、マスタの**表示名と完全一致**しない行を**黙って除外**する。書き込み側（ringo-bsky / `insertFeeds`）は変更しない。

## Considered Options

- **(a) 書き込み時にフィルタ** — テーブルにゴミ行が入らないが、「先に投稿 → 後でマスタ登録」の順になった正当な記録が永久に失われる。register-apple-variety スキルで品種を随時追加する運用と相性が悪い
- **(b) 読み取り時にフィルタ（採用）** — 生データが残るため、後から品種をマスタに登録すると過去の投稿が遡って集計に現れる。マスタの知識（colorMap 等）がすでに集約されている ringo-db の読み取り側に閉じる。代償としてゴミ行が D1 に残り続ける（許容。将来の手動メンテで削除する可能性はある）
- **(c) 未登録名を「その他」に束ねて表示** — 未登録名を隠すという目的と逆行するため不採用

## Consequences

- feeds テーブルにはマスタ未登録の名前が意図的に残る。insert 時に検証がないのはバグではなく、遡及集計のための設計
- 一致条件は表示名との完全一致のみ（正規化なし）。表記ゆれ（空白・半角カナ等）の投稿はグラフに出ないことで投稿者が気づき、投稿側で直す運用とする。色引き（`colorMap.get`）と同じキー・同じ一致条件に揃えることで「フィルタは通るのに色が引けない」不整合を防ぐ
- 未登録名の欠落は観測手段を持たない（個人アプリのため、必要なら D1 への直接クエリで差分を確認する）。「食べたら登録する」運用でカバーする
- フィルタ導入後、色のフォールバック経路（`findColorNameOrUndefined` の `undefined`、`findColorNamePure` の `'red'`）は理論上デッドパスになるが、多層防御として現状維持する。整理する場合はフィルタ稼働後に独立した refactor として行う
