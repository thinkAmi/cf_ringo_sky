# audit-plan-step

実装した AI の自己申告を検証せずに信じない、が出発点。監査は**読み取り専用**で行い、
問題を見つけても直さない(修正は実装セッションの担当)。

## 1. 要求事項の抽出

計画 doc の当該ステップと docs/adr/ から、作業項目・合格条件・禁止事項・順序制約を列挙する。
「計画に書いていないが ADR が要求していること」(API 不変、命名規約など)も含める。

## 2. 機械ゲートの独立再実行

実装セッションの報告値を使わず、自分で実行し直す:
typecheck → `biome check` → ビルド(dry-run)→ `bun test` → `verify.sh`。
テストは件数だけでなく、**計画が要求したテスト設計が実在するか**(例: ルール別違反フィクスチャ、
全祖先ユニークのフィクスチャ、差分テストの撤去条件コメント)を describe/test 名とソースで確認する。

## 3. データ変換の機械照合

変換・転記を伴うステップでは、目視ではなく**照合スクリプトを書いて**突き合わせる。
元データと成果物の全件比較で「欠落 / 値の変更 / 新規追加」を列挙する。
(実例: appleColors.ts ↔ varieties.md の照合で、転記欠落ゼロの証明と同時に、
既存 typo の修正・重複エントリの統合・色の新規付与という3種の無申告差分を検出した)

## 4. コミットメッセージの検査(監査対象にコミットが含まれる場合)

各コミットについて次を確認する:

- **規約準拠(機械)**: CLAUDE.md のコミット規約(ヘッダの正規表現・「なぜ:」行・Co-Authored-By)を満たすか
- **type の妥当性**: 「外から見える挙動が変わるか」の基準と diff が整合するか
  (例: 動作不変が建前のステップに feat が混ざっていたらスコープ逸脱のシグナル)
- **「なぜ」の内容**: 転記元(計画ステップ・完了報告・依頼文)と整合するか。
  diff を見れば分かること(what の再述)だけになっていないか

指摘は「要修正」に分類する。未 push であれば `git commit --amend` で回復可能。

## 5. 分類と報告

結果を2部構成で報告する:

- **想定通りだった内容**: 計画のどの要求に対応するか、根拠(ファイル・テスト名・ゲート結果)つきで
- **想定と異なった内容**: 1件ごとに次のラベルを付ける
  - **正当(把握のみ)**: 計画より正確・忠実な判断。承認時に知っていればよい
  - **要承認**: データの創作・選定など、人間が明示的に認めるべき判断
  - **要修正**: プロセス逸脱(ブランチ運用・スコープ侵食・無承認コミット等)。回復手順も添える

ラベルの判断に迷ったら重い側に倒す。監査で問題ゼロでも「検証した範囲」を明示する
(実行していないゲートがあるなら、その旨を正直に書く)。
