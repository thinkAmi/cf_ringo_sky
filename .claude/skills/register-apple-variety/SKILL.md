# register-apple-variety

品種マスタ `packages/ringo-db/data/varieties.md` に1品種 = 1行を追加・更新するスキル。
**編集するのはこのデータファイルのみ。** バリデータに落とされたらデータ側を直す。

## スコープ

**やること**
- varieties.md への行の挿入(新規)または行の置換(既存品種の更新)
- 機械ゲート `bun test` の実行と、エラーメッセージを読んでのデータ修正
- 完了報告 → ユーザー確認 → 承認後に **commit スキル**で1コミット

**やらないこと(実行しない)**
- パーサ・バリデータ(`packages/ringo-db/src/varietyMaster.ts`)やテストのコード編集。
  lint ルール自体が間違っていると思ったら、回避せず報告して止まる
- push・PR 作成・マージ・デプロイ → **すべて人間が行う**
- 架空の品種・出典のでっち上げ。入力はユーザーから得る

## 前提となるドメイン知識

- 品種マスタは**色と系譜の source of truth**。主キーは表示名(日本語の名前空間)
- **グラフの色は表示名と投稿表記の厳密一致で付く。** Bluesky 投稿のバッククォート内文字列が
  そのまま `feeds.name` になり、品種マスタの表示名と完全一致したときだけ色が引ける。
  全角/半角・スペース・大文字小文字のゆれがあると `red` フォールバックになるため、
  表示名は「投稿でバッククォートに書く正確な表記」をユーザーに確認する
- **系譜は任意。** 持つ品種だけ name(snake_case)・花粉親・種子親を埋め、
  持たない品種はこの3セルを空にする(中間状態は lint が弾く)
- **基本品種(親をたどれない品種)の親は空セルではなく `unknown` を明示する。**
  表示名「不明」の行は書かない(パーサが内部で合成する)
- **読みはソートキーであってふりがなではない。** ひらがな + 長音記号のみ(`^[ぁ-んー]+$`)。
  英字品種もひらがな転写で書く(例: KORU → こる)
- **出典は行の内容全般(系譜・色・読み)の根拠。** 自由記述(Web は markdown リンク、
  書籍は書名等)、空も可。画面・API には出ない保持専用データ
- 本番反映は ringo-db Worker のデプロイ(人間の担当)。このスキルはコミットまで

## 手順

### 1. 入力を確認する

- **表示名**(必須): 投稿でバッククォートに書く正確な表記か、ユーザーに確認する
- **読み**: 確信が持てなければ Web 検索で確認するか、完了報告で「読みは推定」と申告する
- **色**: CSS named color のみ(カラーコード不可。lint が named color の実在を検証する)。
  「赤系」等の系統指定なら、既存の使用色を確認して見分けやすい色を選ぶ:
  ```bash
  awk -F'|' 'NR>2 {gsub(/ /,"",$4); print $4}' packages/ringo-db/data/varieties.md | sort | uniq -c | sort -rn
  ```
- **系譜を登録する場合**: name(snake_case)・花粉親・種子親(不明なら `unknown`)を確認し、
  **出典を尋ねる**。ない場合は完了報告で「出典なし」と明示する

### 2. 前提チェック

花粉親・種子親が varieties.md の name 列に実在するか確認する(`unknown` は常に可)。
未登録の親を参照すると lint(参照整合)で落ちるため、**先にその親の登録を提案する**。

### 3. 行を作って挿入する

- セルは7つ。系譜なしの品種は name・花粉親・種子親を空にする(実在行の例):
  ```markdown
  | 奥州ロマン | おうしゅうろまん | FireBrick | ohshu_roman | tsugaru | shinano_gold | |
  | 王林 | おうりん | YellowGreen |  |  |  | |
  ```
  出典を書く場合は末尾セルに `[タイトル](URL)` または書名等のテキストを入れる
- **読みの五十音順の正しい位置に挿入する**(位置は lint が検証する)。
  既存品種の更新は該当行を置換する(表示名の重複は lint が弾く)
- 出典セル内の半角 `|` は `\|` にエスケープ、複数出典は ` / ` 区切り(いずれも慣例。lint 対象外)

### 4. 機械ゲート

```bash
bun test
```

エラーメッセージには行番号と原因が含まれる(例: `読み順ソートが不正です: 42行目 (期待位置: 38行目)`)。
メッセージを読んで **varieties.md を修正**し、全テストが緑になるまで再実行する。

続けて API スナップショットを確認する(登録内容は API 出力に反映されるため、baseline が古くなる):

```bash
./verify/verify.sh
```

- 差分が出たら、**今回の登録の反映だけか**を確認する(例: 系譜つき品種なら `/api/genealogies` への
  1エントリ追加)。想定内なら `./verify/verify.sh --update` で baseline を再生成し、
  `git diff verify/baseline/` で差分が登録内容と一致することを確かめる
- 再生成した baseline は**品種登録と同じコミットに含める**(次に verify.sh を回すセッションが
  謎の差分に遭遇しないように。実例: ゴールドロマン登録時に baseline が未更新のまま残り、
  後続の PR 3 セッションが想定外の差分を踏んだ)
- 登録した品種と**無関係の差分**が出た場合は、更新せず報告して止まる

### 5. 完了報告 → ユーザー確認

次の4点を含めて報告し、**承認前にコミットしない**:

1. **実施内容**: 追加・更新した行(diff の要点)
2. **ゲート結果**: `bun test` と `./verify/verify.sh` の結果(baseline を再生成した場合はその旨)
3. **判断事項**: 「読みは推定」「出典なし」「色は◯◯系の指定から選定」等。なければ「なし」と明記
4. **人間レビュー依頼**: 機械検証できないデータ(表示名の表記・読み・色の選定・系譜・出典)の確認

### 6. コミット

承認後、**commit スキル**の手順に従って1コミットを作る(1品種の登録 = 1論理単位)。

## このスキルに書かないもの(境界)

- lint ルールの詳細(varietyMaster.ts とそのテストが正。エラーメッセージが案内する)
- コミット規約の詳細(commit スキルと CLAUDE.md の領分)
- デプロイ手順(人間の作業。README の領分)
