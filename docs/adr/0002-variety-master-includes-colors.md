# 系譜 markdown は色定義も含む品種マスタとする

ADR-0001 で導入する markdown を「系譜ファイル」ではなく「品種マスタ」と定義し、`appleColors.ts` にハードコードされていた色定義（129品種）も同じファイルの列として統合する。主キーは表示名（feeds と色定義が使う日本語の名前空間）。snake_case の name と花粉親・種子親の列は系譜を持つ品種だけが埋める。

## Considered Options

- **方法1: 色定義（appleColors.ts）と系譜 markdown を分離したまま、クロスチェックテストで不整合を検出する** — 実装コストは最小だが、品種追加時の2ファイル編集の手間が残る
- **方法2: 品種マスタとして統合（採用）** — 1品種 = 1行になり、色と系譜の不整合が構造的に起こせない。一括変換と add-apple-to-graph スキルの書き換えが一回きり必要

## Consequences

- `findColorName` は markdown をパースした Map を引くようになり、`appleColors.ts` のデータ定義は廃止される
- 現在の `findColorName` は未登録品種を黙って `'red'` にフォールバックしている。統合後、品種マスタ内の色は lint（色名の実在チェック）で検証されるため、フォールバックが働くのは feeds 由来の未登録品種名に対する実行時のみとなる（この経路は実行時データのためテストでは守れず、フォールバック自体は維持する）
- feeds の品種名は実行時に Bluesky 投稿から来るため、「食べた品種が品種マスタに未登録」はビルド時テストでは防げない（従来と同じ性質）
- add-apple-to-graph スキルは廃止し、品種マスタへの行追加・更新（色のみ・系譜のみ・両方）を一手に扱う register-apple-variety スキルに統合する（1ファイル = 1スキルで発動の迷いをなくす）
