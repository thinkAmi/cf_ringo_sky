# plan-dependency-upgrade

依存更新の計画書を8フェーズで作成する。成果物は `docs/YYYY-MM-dependency-upgrade-plan.md` 1本。
過去の実施例: `docs/2026-07-dependency-upgrade-plan.md`（PR #7 で完遂）。

**定常レーンとの境界（ADR 0006）**: minor/patch の定期更新は Dependabot の週次 PR が担うため、このスキルの対象外。このスキルが扱うのは major 更新・依存の撤去/再導入・方針変更を伴う更新。Dependabot が作った major の個別 PR はマージせず、このスキルの計画の入力として使う（計画完了時に当該 PR は自動クローズされる）。

## 過去の計画 doc の扱い（最初に読む）

- 直近の計画 doc からは **①不採用・先送り表 ②ポリシー例外記録 の2節だけ**を読む。他の節は参照しない
- 過去 doc からコピーしてよいのは**理由・判断・再評価条件だけ**。版数・依存一覧・制約などの
  事実は必ず現物（package.json / npm view）から取り直す。**事実は再導出、理由は参照**
- 例外記録に時限解除（minimumReleaseAgeExcludes 等）が残っていれば、解除確認を今回の計画に含める

## Phase 0: 起点と制約の確認（AskUserQuestion で確定）

- 動機: 定期更新 / CVE 対応 / 特定機能。CVE なら緊急度と例外要否が変わる
- 運用の取り決め（計画 doc に記録する）: コミットは誰が作るか / 検証スクリプトは誰が実行するか /
  目視の受け入れ判断は誰か / 既存レッドがある場合のゲート判定方式（変更ファイルのみ等）
- スコープ外の明示（CI・デプロイ・別 issue 化するもの）

## Phase 1: インベントリ（機械的収集）

全 workspace の直接依存（workspace:* 除く）について:

```bash
npm view <pkg> time --json        # 全版のリリース日 → 経過日数を算出
npm view <pkg> engines peerDependencies deprecated
bun audit                          # セキュリティ起点の有無
```

- bunfig.toml の `install.minimumReleaseAge`（秒）を確認し、**公開からその日数を経過した版のみ採用候補**とする
- 計画書に**バージョン採用表**を必須で載せる:

| パッケージ | 現在 | 最新 | 採用版 | リリース日 | 経過日数 | ポリシー | 採用理由 |

- 表には**確認実施日を明記**（経過日数は動く値。実行時に再確認する旨も注記）
- 最新を採用しない場合は理由必須（ポリシーブロック / ペア整合 / 破壊的変更回避 など）

### CVE によるポリシー例外

既知の実害（CVE）＞公開直後リスク（仮説）なら例外を検討。例外時は以下のブロックを計画書に必ず残す:

```markdown
### ポリシー例外記録
- 対象: <pkg>@<ver>（リリース N日前、ポリシー未達）
- 根拠: CVE-XXXX-XXXXX（severity、当リポジトリの該当経路の有無）
- 例外方法: bunfig.toml の install.minimumReleaseAgeExcludes に追加
- リスク緩和: バージョン完全固定（^ を使わない）/ リリースノート・diff 確認
- 解除条件: 公開3日経過で excludes から削除（次回作業時に確認）
```

「CVE はあるが該当経路を使っていないので通常フローで待つ」と判断した場合も、その理由を同じ場所に記録する。

## Phase 2: 撤去候補の洗い出し（更新の前に「消せないか」）

撤去は更新工数そのものを消す（例: MUI 撤去で 2世代分の codemod 移行を回避）。判定基準:

| 判定 | 検出方法 |
|---|---|
| 完全未使用 | grep で import ゼロ |
| デッドコード連鎖 | import 元を辿る |
| プラットフォーム標準で代替可 | 使用機能の棚卸し（例: Drawer→dialog、DataGrid→table） |
| 既存依存と機能重複 | 機能マップ（例: react-query→Router loader キャッシュ） |
| シェル/ランタイムで代替可 | 例: concurrently→シェルスクリプト |
| 生成に置換可 | 例: workers-types→wrangler types |

各撤去候補には**成立根拠と再導入条件**を必ず書く（将来「なぜ無いのか」への回答になる）。

## Phase 3: 横断制約と破壊的変更の事前調査

- 横断制約の一覧化: Node/Bun 最低版、ESM 専用化、ペア版制約（例: drizzle orm/kit）、
  パッケージ改名・非推奨→乗り換え先
- メジャー更新ごとに changelog / migration guide を確認し、**何が壊れるはずかを計画に書く**
- 深い内部パス import（`<pkg>/dist/...`）の有無を grep する。ESM 化で壊れる筆頭。
  見つけたら「公開型へ置換」をステップ内容に含める

## Phase 4: ステップ分割と順序付け

- Step 0 は必ず検証基盤（typecheck 一括化・ゴールデンスナップショット・単体テスト安全網）
- **撤去 → 更新**の順
- **挙動変更とバージョン変更を同一ステップに混ぜない**（問題切り分けの担保）
- 基盤(ランタイム) → ツールチェーン → 個別パッケージ → ビルド基盤 → フレームワーク
  （React 等は依存が減った後の最後に置くと影響最小）
- 最大リスクのステップは単独・後半に置き、それまでに検証基盤を枯らしておく
- 1ステップ = 1コミット。各ステップに: 対象 / 作業内容 / **固有チェック** / 目視要否

## Phase 5: 検証の全体設計（5層表を計画書に載せる）

| 層 | 手段 | 適用範囲 |
|---|---|---|
| 1. 型 | tsc --noEmit（全パッケージ） | 全ステップ |
| 2. lint | biome check | 全ステップ |
| 3. ビルド | build:fe/be、wrangler deploy --dry-run | 全ステップ |
| 4. 実行時 | verify.sh（API ゴールデンスナップショット diff） | 全ステップ |
| 5. 目視 | dev サーバーでクリックスルー | フロント挙動変更ステップのみ |

必須の注意書き: **API diff はフロント破壊を検出できない**。フロント挙動変更ステップは層5必須。

## Phase 6: 不採用・先送り表

検討して**やらなかったこと**を理由・再評価条件つきで表にする。次回計画の起点になる、最も再利用される節。

## Phase 7: 期待される最終状態（測定可能に）

直接依存数の増減・バンドルサイズ・全依存の鮮度などを**数値で**書き、達成判定を可能にする。

## 計画完了時の振り分け

- 計画中に見つけた**恒久的な事実**（罠・制約）は計画 doc に留めず CLAUDE.md「落とし穴」や
  コードコメントへ昇格させる（計画 doc は点時刻の記録であり、事実の置き場ではない）
- 計画が実行完了したら doc 先頭にステータス行を追記する:
  `> ステータス: 完了(PR #N マージ済み、YYYY-MM-DD)`
